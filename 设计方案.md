# 活动照片智能筛选 + 结构化语义描述照片库 · 设计方案

> 本文件是从另一个会话接力过来的完整上下文。**新会话第一步：读完本文件。**
> 用户 James 主要用 Claude Code + Codex 辅助开发，回复用中文，简洁直接。

> **⚠ 2026-06-15 范围修订（以本段为准，覆盖下文旧表述）**：
> 本工程**不生成宣传文案**。最终产物是一个**结构化语义描述照片库**——粗筛精筛出优质照片后，对每张做结构化语义描述，建成一个**独立的 obwiki 式 vault**（自包含、context-scoped），供**后续 agent 渐进式披露检索**调用。文案、海报、推文等下游产物由调用本库的 agent 自行生成，不在本工程范围。
>
> - L2 的角色：从「精筛 + 写文案」改为「精筛 + 结构化语义描述」。
> - 借鉴 obwiki 三闭环：本工程 = 照片库的 **ingest 闭环**（capture→distill→link 映射为 粗筛→描述→建索引/互链）；后续 agent 走 **retrieve 闭环**（L0 句柄→L1 描述→L2 原图 渐进式披露）；**evolve** 为阶段性重打分/补标签/去重。
> - 出入口契约见 [`docs/io-contract.md`](./docs/io-contract.md)。

## 0. 这是什么项目

从**大量活动记录照片**（几千到几万张/次）里，自动**粗筛 → 精筛 → 生成宣传文案**。

核心痛点：人工筛图效率低、主观偏差大；老电脑硬件受限（先按**纯 CPU**跑通，但要兼容小独显 / 大独显 / Mac 四档硬件，只改 config 不改代码）。

结构化语义描述要求（取代旧"文案要求"）：每张精选图产出可检索的结构化字段（场景、主体、标签、氛围、人数等）+ 已有的时间/位置，供后续 agent 检索。**不产出成稿文案**——下游 agent 拿这些结构化信息自行生成文案/海报等。

---

## 1. 核心架构判断（重要）

用户初版想法是 OFA 视觉蕴含(2022) 做粗筛 + LAION aesthetic(2022) 做精筛 + Qwen2.5-VL API 写文案。

**评判：OFA 和 LAION aesthetic 都偏老，效果和维护都不理想。本质上它俩都在回答"这图好不好/合不合适"。** 在纯 CPU 约束下重新分层：

- **L0 元数据筛**（CPU，极快）：损坏图、过小图、过暗/过曝、纯模糊（OpenCV 拉普拉斯方差）直接淘汰。砍掉 30-50% 废片，几乎不花算力。
- **L1 本地粗筛**（CPU，SigLIP 零样本）：文本 prompt 给每张图打"宣传适配度"分（清晰人物/积极氛围/明亮光线 vs 模糊/背影/废片），保留 Top N%。**替代 OFA。**
- **L2 VLM 精筛 + 结构化语义描述一步到位**（API）：只把 L1 选出的少量图发给 VLM，**同时**做"是否真适合入库(带理由)"判断 + 输出结构化语义描述（JSON）。**精筛和描述合并成一次调用，省一半 API 钱。** 通过的图连同描述写入照片库 vault。

关键收益：**几万张里只有几十~几百张真正调到 API**，CPU 老电脑完全扛得住。最终产物是 obwiki 式照片库（见 §6），不是成稿文案。

## 2. 技术栈选型（2026 视角）

- **语言/依赖**：Python 3.11 + `uv` 管依赖
- **元数据**：`Pillow` + `exifread`（EXIF 时间/GPS）+ `opencv-python`（模糊/亮度）
  - 时间：EXIF `DateTimeOriginal` 为主，文件名解析为兜底
  - 位置：`reverse_geocoder`（**纯离线** GPS→地名，保证 CPU 离线可跑）；联网可选 Nominatim
- **L1 粗筛**：**SigLIP 2**（Google 2025，强于 2021 CLIP，中英文都行）走 `transformers`；CPU 用 `onnxruntime` 量化；保底退回 `open_clip`
- **美学分（可选）**：LAION aesthetic predictor 作附加排序信号，**不作硬门槛**
- **L2 VLM**：抽象成 `VLMProvider` 接口（呼应 bizwiki 的 LLMProvider 做法），默认接 Qwen-VL 系 API，可切 InternVL3 / 本地 Ollama
  - 模型默认 Qwen2.5-VL，可升级 Qwen3-VL / InternVL3
- **编排/断点续跑**：不上 Celery。用 **SQLite 当状态机**——每张图一条记录（`stage / score / verdict / caption`），跑挂重跑只处理未完成的。几万张 SQLite 足够。

**核心原则：所有模型都走"接口抽象 + 配置切换"**，四种硬件只改 config 不改代码。

## 3. 目录结构

```
photo-promo/
├── AGENTS.md           # 给 Claude Code / Codex 的操作守则
├── pyproject.toml      # uv 管理
├── config.yaml         # 硬件档位/模型/阈值/prompt 全在这切
├── pipeline.db         # SQLite 状态机(派生,可删重建)
├── src/
│   ├── stage0_meta.py      # EXIF + 模糊/亮度筛
│   ├── stage1_clip.py      # SigLIP 粗筛
│   ├── stage2_vlm.py       # VLM 精筛+结构化语义描述
│   ├── providers/          # VLMProvider 抽象 + 各家实现
│   └── db.py               # SQLite 状态机
├── prompts/            # 粗筛 prompt / 描述 prompt 单独成文件
├── library/            # 【出口】obwiki 式照片库 vault（见 §6）
│   ├── index.md        #   L0 索引：全库句柄一览，可全量扫描
│   ├── photos/         #   每张精选图一个描述条目（frontmatter=L0，正文=L1）
│   └── images/         #   精选原图（L2，复制或软链）
└── tests/fixtures/     # 几张样例图,验链路用
```

每个 stage 独立可单跑、prompt 抽成文件方便反复调、config 集中——对 AI coding 友好，一次只改一个 stage 文件。

## 4. 开发计划（每步可独立验证）

- **Phase 0 · 骨架**（半天）：脚手架 + config.yaml + SQLite schema + 5 张样例图 fixtures。产出：`uv run` 能空跑不报错。
- **Phase 1 · L0 元数据筛**（半天）：EXIF 时间/GPS + OpenCV 模糊亮度 + 离线逆地理编码。产出：DB 里每张图有时间/位置/是否废片。
- **Phase 2 · L1 SigLIP 粗筛**（1 天）：SigLIP 加载(CPU 优先) + 宣传适配度打分 + Top N% 入选。产出：几千张能跑完出排序。
- **Phase 3 · L2 VLM 精筛 + 结构化语义描述**（1 天）：VLMProvider 接口 + 火山方舟 Ark 实现 + 精筛/描述合并 prompt（输出 JSON）。产出：精选图描述写入 SQLite。
- **Phase 4 · 串起来 + 导出照片库 vault**（1 天）：一条命令跑全链路、跑挂能续；把入库的图导出成 `library/`（index.md + photos/ 描述条目 + images/ 原图），并建图间互链。产出：`uv run promo ./photos/` → 一个可被后续 agent 检索的 obwiki 式照片库。
- **Phase 5（可选）· 硬件适配 + evolve**：config 切 Ollama 本地 VLM；阶段性重打分/补标签/连拍去重/补互链（对应 obwiki evolve 闭环）。

## 6. 照片库 vault 设计（obwiki 思路落地）

**定位**：独立、自包含、context-scoped 的 obwiki 式照片库；与 `~/baidu/obwiki` 个人知识库**解耦**，可单独交付给任何 agent。完整出入口契约见 [`docs/io-contract.md`](./docs/io-contract.md)，此处只记设计映射。

**obwiki 三闭环 → 照片库映射**：

| obwiki | 照片库 | 谁负责 |
|---|---|---|
| ingest（capture→distill→link） | 粗筛精筛 → 结构化描述 → 建索引/图间互链 | **本工程** |
| retrieve（L0→L1→L2 渐进披露） | L0 句柄 → L1 描述 → L2 原图 | 后续调用方 agent |
| evolve（重蒸馏/补链/去重） | 重打分、补标签、连拍去重、补互链 | 本工程 Phase 5 / 人在环 |

**渐进式披露三层**（北极星，物理强制）：

| 层 | 照片库里是什么 | 何时加载 |
|---|---|---|
| **L0 索引** | `library/index.md` + 每条 frontmatter：`id / path / summary 一句话 / scene / tags / taken_at / location` | 可全量扫描（几十 token/条） |
| **L1 描述** | `library/photos/<id>.md` 正文：结构化语义描述（场景/主体/氛围/人数/适用场景…） | 候选命中后按需读 |
| **L2 原图** | `library/images/<id>.<ext>` | 反思门控选中后才打开（VLM 复看 / 真要用图时） |

**继承的 obwiki 纪律**：① 链接密度北极星——同活动/同场景/同人物的图互链，带"为什么相关"；② 入库四操作 ADD/UPDATE/MERGE/NOOP（连拍去重判定），DELETE 只提议；③ frontmatter 即 L0，Obsidian 可直接浏览。
