## 0. 这是什么项目

从**大量活动记录照片**（几千到几万张/次）里，自动**粗筛 → 精筛 → 生成宣传文案**。

核心痛点：人工筛图效率低、主观偏差大；老电脑硬件受限（先按**纯 CPU**跑通，但要兼容小独显 / 大独显 / Mac 四档硬件，只改 config 不改代码）。

文案要求：包含**时间、位置、场景细节**，例如"2026年6月15日于北京国际会展中心，参会者在互动体验区积极交流"。风格先跑通 demo，后调。

---

## 1. 核心架构判断（重要）

用户初版想法是 OFA 视觉蕴含(2022) 做粗筛 + LAION aesthetic(2022) 做精筛 + Qwen2.5-VL API 写文案。

**评判：OFA 和 LAION aesthetic 都偏老，效果和维护都不理想。本质上它俩都在回答"这图好不好/合不合适"。** 在纯 CPU 约束下重新分层：

- **L0 元数据筛**（CPU，极快）：损坏图、过小图、过暗/过曝、纯模糊（OpenCV 拉普拉斯方差）直接淘汰。砍掉 30-50% 废片，几乎不花算力。
- **L1 本地粗筛**（CPU，SigLIP 零样本）：文本 prompt 给每张图打"宣传适配度"分（清晰人物/积极氛围/明亮光线 vs 模糊/背影/废片），保留 Top N%。**替代 OFA。**
- **L2 VLM 精筛 + 文案一步到位**（API）：只把 L1 选出的少量图发给 VLM，**同时**做"是否真适合宣传(带理由)"判断 + 生成文案。**精筛和文案合并成一次调用，省一半 API 钱。**

关键收益：**几万张里只有几十~几百张真正调到 API**，CPU 老电脑完全扛得住。

## 2. 技术栈选型（2026 视角）

- **语言/依赖**：Python 3.11 + `uv` 管依赖
- **元数据**：`Pillow` + `exifread`（EXIF 时间/GPS）+ `opencv-python`（模糊/亮度）
  - 时间：EXIF `DateTimeOriginal` 为主，文件名解析为兜底
  - 位置：`reverse_geocoder`（**纯离线** GPS→地名，保证 CPU 离线可跑）；联网可选 Nominatim
- **L1 粗筛**：**SigLIP 2**（Google 2025，强于 2021 CLIP，中英文都行）走 `transformers`；CPU 用 `onnxruntime` 量化；保底退回 `open_clip`
- **美学分（可选）**：LAION aesthetic predictor 作附加排序信号，**不作硬门槛**
- **L2 VLM**：抽象成 `VLMProvider` 接口（呼应 bizwiki 的 LLMProvider 做法），默认接 Qwen-VL 系 API，可切 InternVL3 / 本地 Ollama
  - 模型默认 Qwen2.5-VL，可升级 Qwen3-VL / InternVL3
- **编排/断点续跑**：不上 Celery。用 **SQLite 当状态机**——每张图一条记录（`stage / score / verdict / caption`），跑挂重跑只处理未完成的。几万张 SQLite 足够。

**核心原则：所有模型都走"接口抽象 + 配置切换"**，四种硬件只改 config 不改代码。

## 3. 目录结构

```
photo-promo/
├── AGENTS.md           # 给 Claude Code / Codex 的操作守则
├── pyproject.toml      # uv 管理
├── config.yaml         # 硬件档位/模型/阈值/prompt 全在这切
├── pipeline.db         # SQLite 状态机(派生,可删重建)
├── src/
│   ├── stage0_meta.py      # EXIF + 模糊/亮度筛
│   ├── stage1_clip.py      # SigLIP 粗筛
│   ├── stage2_vlm.py       # VLM 精筛+文案
│   ├── providers/          # VLMProvider 抽象 + 各家实现
│   └── db.py               # SQLite 状态机
├── prompts/            # 粗筛 prompt / 文案 prompt 单独成文件
├── out/
│   ├── selected/       # 精选图(软链/复制)
│   └── captions.md     # 文案输出
└── tests/fixtures/     # 几张样例图,验链路用
```

每个 stage 独立可单跑、prompt 抽成文件方便反复调、config 集中——对 AI coding 友好，一次只改一个 stage 文件。

## 4. 开发计划（每步可独立验证）

- **Phase 0 · 骨架**（半天）：脚手架 + config.yaml + SQLite schema + 5 张样例图 fixtures。产出：`uv run` 能空跑不报错。
- **Phase 1 · L0 元数据筛**（半天）：EXIF 时间/GPS + OpenCV 模糊亮度 + 离线逆地理编码。产出：DB 里每张图有时间/位置/是否废片。
- **Phase 2 · L1 SigLIP 粗筛**（1 天）：SigLIP 加载(CPU 优先) + 宣传适配度打分 + Top N% 入选。产出：几千张能跑完出排序。
- **Phase 3 · L2 VLM 精筛+文案**（1 天）：VLMProvider 接口 + 一家 API 实现 + 精筛/文案合并 prompt。产出：精选图 + captions.md。
- **Phase 4 · 串起来 + 断点续跑**（半天）：一条命令跑全链路，跑挂能续。产出：`uv run promo ./photos/`。
- **Phase 5（可选）· 硬件适配 + 调风格**：config 切 Ollama 本地 VLM、文案风格 prompt 迭代。
